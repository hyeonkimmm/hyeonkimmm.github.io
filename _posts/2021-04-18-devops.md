## AWS 와 Docker

### AWS
1. 장점
    - 탄력적인 웹 규모 컴퓨팅
    - 다양한 API 사용
        - 빠르게 추가 삭제 가능
    - 유연한 클라우드 호스팅
    - 통합된 서비스
    - 안정성
    - 보안
2. 단점
    - 베어 메탈 성능을 원할 때
    - 웹페이지가 몇개뿐일 때
    - 솔루션에 적합
    - 가격(잘 사용하지 않으면 비쌈)
    -> 작은서비스 가 아니면 좋음

### AWS의 종류
![dev_process](/image/lecture/aws_kind.png)
1. Server
    1. EC2(Elastic Compute Cloud)
        - 안전
        - 크기 조정
        - 클라우드에서 제공하는 웹 서비스
        - 장점
            - 자동 확장 및 축소
            - 완전 제어
            - 다양한 커맨드 api 제공
            - S3, RDS 통합성
            - 안정성 보안
    2. Lambda
        - 프로비저닝 관리 필요 X
        - 사용한 컴퓨팅 시간만큼 비용 지불
        - 모든 유형의 app , backend service 별도의 관리없이 실행 가능
        - 높은 가용성
        - 장점
            - aws service 쉬운 확장
            - S3 , DynamoDB custom api 가능
    3. VPC(Virtual Private Cloud)
        - 가상 네트워킹 환경을 완벽하게 재현
        - 장점
            - 다중 연결 옵션
            - 사용자 제어
            - 보안
2. Storage
    1. S3
        - 쿼리 지원 기능
        - DB에 대한 강력한 분석 실행
        - 장점
            - 높은 안정성
            - 높은 확정성
            - 데이터 자동 복제
            - 민감한 데이터 자동 분류
    2. EBS(Elastic Block Store)
        - AWS
        - snapshot 기능
3. DataBase
    1. RDS
        - 다양한 DB Engine 제공
        - 사용의 편의성
            - 다양한 api 제공
        - DB를 세부적으로 제어, 튜닝
        - SSD storage
    2. DynamoDB
        - 10m/s 미만
        - 완전 관리형 cloud service
        - 기능
            - 초당 수백만개 요청 micro 성능 제공
    3. RedShift
        - 속도가 빠른 완전 관리형 DB
        - 간편 , 비용의 효율적
        - 정교한 쿼리, 최적화
        - 고성능, 대량 데이터 처리 가능
        - SQL 작업 병렬처리
        - local 연결 처리
        - 아무튼 참 좋다
    4. AuroraDB
        - 가용성 안정성 보안성 1/10 비용
        - 속도 빠름
        - storage auto scaling
        - 읽기전용 복제 가능
4. Management
    1. Cloud Watch
        - 로그 파일 수집
        - 모니터링
        - 자동 대응
        - EC2 instance AWS 리소스 뿐 아니라 다양한 app의 모든 로그파일 모니터링 가능
        - 성능 , 운영상의 상태 파악 가능
        - 비용 추가 없이 사용 가능
        - 비용 추가시 더욱 좋음
5. Analysis
    1. Kinesis
        - 실시간 데이터 수집
        - 완전 관리형
        - 확장 가능
        - 모든 규모의 streaming 가능
### Docker의 장점
- 장점
    - 실행시점에 상관없이 구성 시점을 고를 수 있음(눈송이 서버)
    - 동일한 이미지를 컨테이너에 배포하게 되면 해당 문제가 많이 줄어듬
    - 프로그램의 설치와 삭제가 용이
    - 도커 실행 소스 일관성 , 유연성
        - 이미지화 한 repo를 통해 동시 작업 가능
        - 서버별로 일관된 세팅 가능
    - 이미지 용량이 크게 줄어듬
        - linux container 사용
    - 여러군데 배포할 수 있는 확장성
        - github과 비슷한 느낌으로 push/pull 가능
